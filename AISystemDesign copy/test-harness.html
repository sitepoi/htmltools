🔀 Cross-App Memory Architecture
🔧 How It Works — Step by Step
1️⃣ Collect
Every app (Ephorm, Copilot, VS Code, web chat) sends conversation data to the Memory API. What was asked? What was the context? What was the answer?
2️⃣ Extract
The Memory System processes each interaction: extracts facts ("Client XYZ revenue is $50M"), preferences ("User prefers bullet points"), entities, and relationships.
3️⃣ Store
Structured memories are saved to Redis/Mem0 as key-value pairs + vector embeddings. Memories are tagged by user, app, topic, and time — searchable and retrievable.
4️⃣ Retrieve & Embed
When ANY app sends a new query, the Memory API retrieves relevant memories and embeds them in the prompt. The AI now has full context — regardless of which app you are using.

💻 GPU Hardware Guide — Local LLM
How trillion-parameter models run on consumer GPUs — MoE, quantization, and the real math
🧠 The Key Concept: Why a $2,000 GPU Can Run a "Trillion Parameter" Model
🔹 Dense Models (Traditional)
All parameters are active on EVERY token. If a model has 70B parameters, all 70B are loaded in VRAM and used for every word generated.

Example: Llama 3.3 70B = 70B active params.
VRAM at 4-bit: 70B × 0.5 bytes ≈ 35 GB → needs A100.
🔹 MoE — Mixture of Experts (Modern)
Only a SMALL FRACTION of parameters are active per token. The model has many "experts" but only routes each token to 1-3 of them. Total params can be huge — but VRAM depends on ACTIVE params, not total.

Example: DeepSeek V4 Pro = 1.6 TRILLION total but only 49 BILLION active.
VRAM at 4-bit: 49B × 0.5 bytes ≈ 25 GB → fits on RTX 5090! 🎉
🔢 Quantization — The 4x VRAM Multiplier
FP16 (full precision): 2 bytes per parameter. Best quality. Used for training.
4-bit (quantized): ~0.5 bytes per parameter. 4× less VRAM. Tiny quality loss. Used for inference.
Rule of thumb: Active params × 0.5 = VRAM needed at 4-bit. Active params × 2 = VRAM at FP16.
⚠️ "Do I need to load ALL 1.6 trillion parameters somewhere?" — YES. Here's the full picture:

Storage Hierarchy for MoE Models:
🗄️ SSD/Disk: Full model stored here. DeepSeek V4 Pro at 4-bit = ~800 GB on disk.
🧠 System RAM (CPU): Frequently used experts cached here for fast access. 64-128 GB RAM helps.
⚡ GPU VRAM: Only the 49B active experts for the current token sit here. ~25 GB needed.

How It Actually Works — "Expert Offloading":
1. The model files (~800 GB) live on your SSD.
2. When you start the model, frequently used "core" layers load into GPU VRAM.
3. As each token is processed, the router decides which experts to activate.
4. Needed experts are prefetched from RAM/SSD into VRAM before they're used.
5. Unused experts are swapped out — they don't stay in VRAM.
6. Only ~49B parameters are computed per token, but all 1.6T are accessible via the storage hierarchy.

This is why you need:
• GPU VRAM: Enough for active experts (~25 GB for DeepSeek V4 Pro at 4-bit)
• System RAM: Helps cache experts — 64 GB recommended for smooth performance
• Fast SSD: NVMe SSD for expert swapping — SATA SSD is too slow
• The result: DeepSeek V4 Pro (1.6T total) runs on RTX 5090 (32 GB VRAM) + 64 GB system RAM + NVMe SSD at ~48 tok/s

🔑 Key insight: "49B active" doesn't mean the other 1.55T parameters vanish — they live on your SSD and in system RAM, ready to be swapped into the GPU when the router calls for them. The GPU only COMPUTES 49B per token, but all 1.6T must be STORED and ACCESSIBLE.
⚡ Performance Deep-Dive: What Happens When Every Question Uses Different Experts?

The concern: If each query activates completely different experts, won't constant loading/unloading kill performance?

The reality — three techniques make it work:

1. 🧠 Expert Prefetching (like CPU branch prediction, but for AI)
The inference engine doesn't wait until it needs an expert. The router predicts which experts will be needed next and prefetches them from RAM to VRAM before the computation reaches that layer. By the time the GPU is ready to compute, the expert is already there. This hides most of the latency.

2. 🔥 Expert Caching — The 80/20 Rule
Expert usage follows a power law. Some experts handle general knowledge ("What is IFRS 16?") and get used constantly. These "hot" experts stay permanently cached in VRAM. Cold experts (specialized topics) may swap in/out, but they're rarely needed. ~20% of experts handle ~80% of tokens.

3. 📦 Batched & Overlapped I/O
While the GPU computes one layer, the engine simultaneously loads the next layer's experts from RAM. This compute/IO overlap means the GPU rarely sits idle waiting for data. NVMe SSDs (~7 GB/s) can load a 4-bit expert (~200 MB) in ~30ms — but this happens in parallel with computation, not after it.

Real-World Performance Numbers (DeepSeek V4 Pro on RTX 5090):
• 64 GB system RAM: ~48 tok/s — most experts cached in RAM, minimal SSD access
• 32 GB system RAM: ~35 tok/s — more SSD reads, but prefetching hides most latency
• 16 GB system RAM: ~20 tok/s — heavy SSD thrashing, noticeable slowdown on diverse queries

🎯 The practical takeaway: For a presentation or Q&A session where questions vary, you'll get ~48 tok/s with adequate RAM (64 GB). The system is designed for this — it's not like loading a new app every time. The router, prefetcher, and cache work together to make expert switching nearly invisible. The $300 extra for 64 GB instead of 32 GB RAM is the best performance investment you can make.

💾 Prompt Caching — What Actually Gets Saved (and What Doesn't)
⚠️ You're right — every parameter IS loaded, and every NEW token IS fully processed. Caching does NOT skip neural network layers.
So what DOES get saved? The answer is the KV-Cache (Key-Value Cache) — a specific part of the attention mechanism.
🔬 What the KV-Cache Actually Saves

Every transformer layer computes attention between all pairs of tokens: "How much should token 5 pay attention to token 1, token 2, token 3, token 4?" This is O(n²) — expensive.

For each token, the model computes a Key (K) and Value (V) vector at every layer. These K,V vectors are stored in the KV-cache.

When the next request starts with the EXACT same prefix (system prompt):
• The K,V for those cached tokens are already computed — skip the matrix multiplications for attention
• New tokens still go through ALL layers fully
• All parameters are still loaded in VRAM — nothing changes there

What's saved: Attention computation (Q×K^T) for the cached prefix — this is the O(n²) part. For a 10K token system prompt, this is ~100M operations skipped per layer.
What's NOT saved: Forward pass through feed-forward layers for new tokens. Parameter loading. Output generation.
☁️ Cloud Pricing: Why Cached Tokens Cost 90% Less

Cloud providers charge less for cached tokens because they save real GPU compute — not just networking:

OpenAI: 90% discount on cached input (GPT-5.4: $2.50 → $0.25/M tok)
Anthropic: 90% discount (Claude Opus 4.8: $5.00 → $0.50/M tok)
DeepSeek: 98% discount! ($0.14 → $0.0028/M tok cache hit)

Real savings example: Enterprise app with 10K token system prompt, used 1,000 times/day = 10M tokens/day that are cached. Without cache: $1.40/day. With DeepSeek cache: $0.028/day. That's $500/yr vs $10/yr.

🖥️ Local KV-Cache: Built into every inference engine (Ollama, vLLM, llama.cpp). Saves real GPU cycles — lets you serve more users with the same GPU. Since local is fixed-cost, it doesn't "save money" directly, but it increases throughput (more queries/sec on the same hardware).
💡 Design for cache hits: Put static content at the BEGINNING of every prompt (system prompt, tool definitions, RAG templates). Put dynamic content at the END (user query, retrieved documents). This maximizes the prefix that can be cached. In enterprise apps, 80-95% of input tokens can be cacheable.
🤔 Why Can Cloud Be Cheaper Than Local? (The Utilization Answer)
The core reason: GPU UTILIZATION. Your RTX 5090 sits idle ~90% of the time (you query it for a few seconds, then it waits). Cloud GPUs run 24/7 serving thousands of users. A GPU that runs at 10% utilization costs 10× more per token than one at 100% utilization — even if the hardware is cheaper.
1. 📊 Utilization Economics
Your RTX 5090 ($2,500):
• 5 users, 15 queries each = 75 queries/day
• Each query: ~3 seconds of GPU time
• GPU active: ~4 min/day = 0.3% utilization
• Cost per token: very high because the GPU sits idle

Cloud H100 ($30,000):
• 500 users, 50 queries each = 25,000 queries/day
• Continuous batch processing
• GPU active: 24h/day = ~95% utilization
• Cost per token: very low because the GPU never sleeps
2. ⚡ Hardware Efficiency
• H100: ~150 tok/s at FP16, 700W
• RTX 5090: ~48 tok/s at 4-bit, 575W
• H100 produces 3× more tokens per watt
• Data center electricity: $0.06/kWh (industrial)
• Home/office electricity: $0.12/kWh (residential)
• Cloud providers get bulk hardware discounts (30-50% off list)
3. 📦 Batch Processing
• One H100 serves 50-100 users simultaneously via continuous batching
• Your RTX 5090 serves 1-5 users sequentially
• Batching amortizes the fixed cost of loading parameters across many requests
• DeepSeek specifically: their infrastructure efficiency is industry-leading, which is why they can charge $0.14/M tok at a profit
4. 🔄 The Break-Even Reality
When local DOES beat cloud:
• 50+ users, 100+ queries/day each
• GPU runs at 60%+ utilization
• Sensitive data that CANNOT leave premises
• Latency-critical applications (local = no network round-trip)

When cloud wins:
• Low-to-medium volume (under 500M tok/mo)
• Variable/spiky workloads
• Need latest models (GPT-5.4, Claude 4.8)
• Zero DevOps overhead needed

• Per minute: 48 × 60 = 2,880 tokens

• Per month: 4.1M × 30 = ~124 million tokens at 100% utilization 24/7
