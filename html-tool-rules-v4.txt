── AI Prompt Relay ────────────────────────────────────
tool.requestAI(prompt, context, callback)

Sends a prompt to the CMS AI service and returns the response text.

  • prompt   — string, the main prompt/question to send
  • context  — string (optional), supporting background information that
               helps the AI understand the situation
  • callback — function(error, responseText)
      error        — null on success; string describing the problem on failure.
                     On truncation: error contains a warning but response is
                     still returned. Example: "Prompt was truncated from
                     150,000 to 100,000 characters."
                     Always check response even when error is non-null.
      responseText — the AI's text response, or null on complete failure

Limits: 60 second timeout per request.
        3 concurrent requests maximum (additional requests are queued).
        100K character prompt limit (prompts exceeding this are truncated;
        check the error param for a truncation notice).
Requires: allowAi: 'yes' in toolParams.

Example:
  tool.requestAI('Summarize this data', JSON.stringify(tableData),
    (err, response) => {
      if (err) { tool.notify('AI error: ' + err, 'warning') }
      if (response) showSummary(response)
    })

── File Upload ────────────────────────────────────────
tool.requestUpload(accept, callback)

Opens the host file picker dialog, uploads the selected file to the CMS

  • accept   — string, MIME types or extensions filter.
               Examples: '.pdf,.docx,.txt'  or  'image/*'  or  '.csv'

  })

── PDF Export ─────────────────────────────────────────  ★ NEW
tool.requestExportPdf(options, callback)

Generates a printable HTML document from the provided content, uploads it to
CMS Storage, and returns the file URL. The user can open it in a browser tab
and use Print → Save as PDF to create the final PDF.

  • options — { html?, filename?, landscape? }
      html      — string, HTML body content to export. If omitted, the system
                  uses the tool's current DOM body content.
      filename  — string, base filename without extension (default: 'export').
                  The returned file will be named like 'export.html'.
      landscape — boolean, hint for print orientation. Informational only;
                  the user selects orientation in the print dialog.
  • callback — function(error, file)
      error — null on success, or string describing the problem
      file  — { name, url, size, type } on success.
              name: filename.html
              url:  publicly accessible URL (open in new tab to print)
              size: HTML document size in bytes
              type: 'text/html'
Requires: allowExportPdf: 'yes' in toolParams.

Example:
  tool.requestExportPdf({
    html: '<h1>Monthly Report</h1><table>...</table>',
    filename: 'monthly-report'
  }, (err, file) => {
    if (err) { tool.notify('Export failed: ' + err, 'error'); return }
    tool.notify('Report exported: ' + file.url, 'success')
    window.open(file.url, '_blank')  // user prints from here
  })

  // Export current tool DOM content:
  tool.requestExportPdf({ filename: 'data-export' }, (err, file) => {
    if (!err) window.open(file.url, '_blank')
  })

── Send Email ─────────────────────────────────────────  ★ NEW
tool.requestSendEmail(options, callback)

Sends an email through the CMS email system (AWS SES / nodemailer). The body
content you provide is automatically wrapped in the standard branded email
template with header, footer, and borders — you only provide the inner body HTML.

  • options — { to, subject, htmlBody, title?, attachments? }
      to          — string, recipient email address (required)
      subject     — string, email subject line (required)
      htmlBody    — string, HTML body content — gets wrapped in the branded
                    template automatically (required)
      title       — string, optional large title shown in the email header.
                    Defaults to the subject if omitted.
      attachments — array of { filename, url }, files to attach.
                    Use tool.requestUpload() first to get file URLs, then
                    pass them here. Each attachment: { filename: 'report.pdf',
                    url: 'https://storage.../report.pdf' }
  • callback — function(error, result)
      error  — null on success, or string describing the problem
      result — { ok: true } on success
Requires: allowSendEmail: 'yes' in toolParams.
The CMS must have email sender configured in website settings.

Example:
  tool.requestSendEmail({
    to: 'customer@example.com',
    subject: 'Your Invoice #INV-0042',
    title: 'Invoice',
    htmlBody: '<h1>Invoice</h1><p>Thank you for your order. Total: $447.95</p>'
  }, (err, result) => {
    if (err) { tool.notify('Email failed: ' + err, 'error'); return }
    tool.notify('Email sent!', 'success')
  })

  // With attachment — upload first, then attach:
  tool.requestUpload('.pdf', (err, file) => {
    if (err) return
    tool.requestSendEmail({
      to: 'client@example.com',
      subject: 'Your Report',
      htmlBody: '<p>Please find the report attached.</p>',
      attachments: [{ filename: file.name, url: file.url }]
    }, (err2, result) => {
      tool.notify(err2 ? 'Email failed' : 'Sent with attachment',
        err2 ? 'error' : 'success')
    })
  })

── Object CRUD (Multi-Type CMS Database Access) ───────
tool.requestObjects(action, params, callback)

Performs CRUD operations on CMS objects across ANY mainObjectType the tool
is authorized for. This allows you to build full applications (product databases,
inventory systems, invoice workflows, ERP modules, etc.) with the CMS as your
database, using a completely custom UI you design.

Each CMS object is a Firestore document with up to 1MB of data. You can store
unlimited objects across unlimited types.

  • action   — string: 'query' | 'get' | 'create' | 'update' | 'delete' | 'batch'
  • params   — object, varies by action (see below)
  • callback — function(error, result)
      error  — null on success, or string describing the problem
      result — varies by action (see below)

── Actions ───────────────────────────────────────────────────────────────────

  query:  { mainObjectType, typeId? }
          • typeId is optional — omit to query all folders of that type
          result: { objects: [...] } — array of full CMS objects

  get:    { mainObjectType, objectId }
          result: { object: {...} } — single CMS object or null if not found

  create: { mainObjectType, typeId?, name, productData: { data_categoriesBased: {...} } }
          • typeId is optional — omit to use the default auto-created folder
          • name: human-readable name for the object
          • productData.data_categoriesBased: { fieldId: value, ... }
            fieldId must match a field defined in the CMS for this type
          result: { object: {...} } — the newly created CMS object

  update: { mainObjectType, objectId, productData: { data_categoriesBased: {...} }, name? }
          • Only include fields you want to change — partial updates are supported
          • Omit productData to only update the name, or vice versa
          result: { ok: true }

  delete: { mainObjectType, objectId }
          • This permanently deletes the object — no undo
          result: { ok: true }

  batch:  { operations: [{ action, mainObjectType, typeId?, objectId?, name?, productData? }, ...] }
          • Execute multiple operations across ANY authorized types in a single
            atomic transaction
          • All operations succeed or all fail (Firestore writeBatch guarantee)
          • Max 500 operations per batch
          • Each operation has the same shape as its individual action
          result: { ok: true, results: [...] } — array of per-operation results

── Object shape ──────────────────────────────────────────────────────────────
Each CMS object returned by query/get/create has this structure:

  {
    id, _id, contentId,          // identity fields (id is the primary key)
    name, slug,                   // display fields (name is human-readable)
    cmsObjectType,                // the mainObjectType string
    typeId,                       // folder/category ID the object belongs to
    tenantId,                     // multi-tenant scoping (automatic)
    productData: {                // the actual configurable data
      data_categoriesBased: {     // field values keyed by field ID
        fieldId1: value1,         // value type depends on field definition
        fieldId2: value2,         // can be string, number, boolean, array, object
        ...
      }
    },
    _parentObjectId,               // parent CMS record ID (for 'instance' scope)
    permissions: { ... },         // access control (automatic)
    created, updated              // ISO timestamp strings
  }

── Authorization ─────────────────────────────────────────────────────────────
The CMS admin must configure two settings on the html-tool field:

  settings.allowObjectCRUD: 'yes'           ← enables tool.requestObjects()
  settings.allowedObjectTypes: [            ← which CMS types the tool can access
    { mainObjectType: 'taskSprints-uniconbaseapps', role: 'editor', scope: 'shared' },
    { mainObjectType: 'taskItems-uniconbaseapps',   role: 'editor', scope: 'instance' },
  ]

  • 'viewer' role — only query and get are allowed; write operations blocked
  • 'editor' role — all actions including create, update, delete, batch
  • scope: 'shared' (default) — data visible across ALL instances of this tool
  • scope: 'instance' — data isolated to the specific CMS record. Query
    auto-filters by the parent object; get/update/delete verify ownership.
    All created objects are automatically stamped with _parentObjectId.
  • mainObjectType can be any string — types do NOT need to be pre-registered
    in CMS settings. The bridge auto-creates a default folder on first use.
  • All operations go through CMS permissions automatically. You don't need
    to manage permissions yourself.
  • Data is stored in om_private_objects and is visible in the
    standard ObjectManager UI if the type is registered in CMS Settings.

── Examples ──────────────────────────────────────────────────────────────────

  // Query all products in a specific folder
  tool.requestObjects('query', {
    mainObjectType: 'products',
    typeId: 'folder-electronics'
  }, (err, result) => {
    if (err) { tool.notify(err, 'error'); return }
    result.objects.forEach(p => {
      const d = p.productData.data_categoriesBased
      addTableRow({ id: p.id, name: p.name, sku: d.sku, price: d.price })
    })
  })

  // Get a single object by ID
  tool.requestObjects('get', {
    mainObjectType: 'warehouses',
    objectId: 'wh-main'
  }, (err, result) => {
    if (err) { tool.notify(err, 'error'); return }
    showDetail(result.object)
  })

  // Create a new invoice
  tool.requestObjects('create', {
    mainObjectType: 'invoices',
    name: 'Invoice #INV-0042',
    productData: {
      data_categoriesBased: {
        customerId: 'cust-acme',
        items: [
          { productId: 'prod-widget', quantity: 5, unitPrice: 29.99 },
          { productId: 'prod-gadget', quantity: 2, unitPrice: 149.00 }
        ],
        totalAmount: 447.95,
        status: 'issued'
      }
    }
  }, (err, result) => {
    if (err) { tool.notify(err, 'error'); return }
    tool.notify('Invoice ' + result.object.name + ' created', 'success')
    addInvoiceToTable(result.object)
  })

  // Update an existing object (partial update)
  tool.requestObjects('update', {
    mainObjectType: 'inventoryCounts',
    objectId: 'inv-001',
    productData: { data_categoriesBased: { quantity: 95 } }
  }, (err, result) => {
    if (err) { tool.notify(err, 'error'); return }
    tool.notify('Stock updated', 'success')
  })

  // Delete an object
  tool.requestObjects('delete', {
    mainObjectType: 'products',
    objectId: 'prod-obsolete'
  }, (err, result) => {
    if (err) { tool.notify(err, 'error'); return }
    removeRow('prod-obsolete')
    tool.notify('Product deleted', 'info')
  })

  // Batch: create invoice + decrement stock in one atomic transaction
  tool.requestObjects('batch', {
    operations: [
      { action: 'create', mainObjectType: 'invoices', name: 'INV-0043',
        productData: { data_categoriesBased: { customerId: 'cust-1', total: 300, status: 'issued' } } },
      { action: 'update', mainObjectType: 'inventoryCounts', objectId: 'inv-001',
        productData: { data_categoriesBased: { quantity: 95 } } },
      { action: 'update', mainObjectType: 'inventoryCounts', objectId: 'inv-002',
        productData: { data_categoriesBased: { quantity: 48 } } }
    ]
  }, (err, result) => {
    if (err) {
      tool.notify('Transaction failed — all operations rolled back: ' + err, 'error')
      return
    }
    tool.notify('Invoice created and stock adjusted', 'success')
  })

NAMING METHODOLOGY (CRITICAL)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

── NAMESPACE CONVENTION ───────────────────────────────────────────────────────

  SHARED (default):      <typeName>-<developerSubdomain>
      e.g. taskSprints-uniconbaseapps, taskItems-uniconbaseapps
      Accessible to ALL tools and apps published by this developer.
      Use this when multiple apps need to share the same data.

  ISOLATED:              <toolId>.<typeName>-<developerSubdomain>
      e.g. ptm.taskSprints-uniconbaseapps
      Private to one specific tool. The toolId prefix (dot-separated) 
      prevents accidental cross-tool access. Use this when data must
      stay isolated — e.g. internal tool state, draft-only data.

  • The developer subdomain is ALWAYS required (matches app-store convention).
  • The dot separates the isolation prefix from the type name.
  • Never use bare names like 'products' or 'tasks' — they pollute
    the global namespace and will collide with other developers.

── GRADUATION PATH ────────────────────────────────────────────────────────────

  When a tool-created type matures and needs a dedicated management UI:

  1. Register it in CMS Settings (App Designer) under the SAME name.
     e.g. register taskSprints-uniconbaseapps as-is — no rename needed.
  2. Define its field groups and fields.
  3. Existing data persists — the type name doesn't change.
  4. It now appears in ObjectManager, reporting, and the catalog.

  Shared types can graduate without migration.
  Isolated types (toolId prefixed) should stay isolated — if you later
  want to share, migrate data to a shared type name first.

── DISCOVERY ──────────────────────────────────────────────────────────────────

  To see all types created by html-tools (including unregistered ones):
  • Open App Designer → "Unregistered Types" panel
    This lists every mainObjectType found in om_objectTypes that isn't
    yet registered in CMS Settings, with a one-click Register button.
  • Or query directly: check om_objectTypes_<tenant> collection —
    each document's mainObjectType field tells you which type it belongs to.

── RULES ──────────────────────────────────────────────────────────────────────

1. ALWAYS namespace new types: <typeName>-<developerSubdomain>.
   Add <toolId>. prefix for isolated types.
2. Use EXISTING mainObjectType values from the catalog above whenever possible.
3. Use EXACT field IDs from the catalog — never guess them.
4. Document new types in settings.allowedObjectTypes so the admin can
   audit which tools own which types.
5. Prefer reusing existing shared types from your developer namespace.
6. New types auto-create a default folder on first use via _ensureObjectFolder.
7. Default folder is named "Default" in om_objectTypes_<tenant> — this is
   what makes the type discoverable in the Unregistered Types panel.
8. Use scope: 'shared' for data shared across instances (e.g. sprints,
   shared definitions). Use scope: 'instance' for data private per
   CMS record (e.g. tasks, private notes, user-specific items).

tool.onReady((val, fields) => {
  render(val, fields)               // initial draw — value + all sibling fields
  if (tool.isReadOnly()) lockUI()   // read-only may be active on first load

  tool.onValueChange(v  => render(v))
  tool.onFieldsChange(f => syncFields(f))
  tool.onReadonlyChange(ro => lockUI(ro))
})

Replace render / syncFields / lockUI with your actual functions.
Never skip isReadOnly() inside onReady — the form can be in read-only
mode on the very first load.
Always define tool.onReady as your single entry point — never use
DOMContentLoaded, window.onload, or inline onload attributes.

HARD RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✗  No <script>, <style>, <html>, <head>, <body>, <!DOCTYPE> tags in any field
✗  No DOMContentLoaded or window.onload — use tool.onReady instead
✗  No localStorage or sessionStorage
✗  No fetch to external URLs (sandboxed: allow-scripts, allow-forms, allow-downloads)
✗  No alert(), confirm(), or prompt()
✗  No raw postMessage calls — always use the tool.* SDK
✓  External CDN <script> tags ARE allowed inside the HTML field
   (e.g. Chart.js, Sortable, DataTables, SheetJS, etc.)
✓  Values can be any JSON type — save objects/arrays freely with tool.setValue()
✓  tool.requestAI / tool.requestUpload / tool.requestFileContent /
   tool.requestObjects / tool.requestExportPdf / tool.requestSendEmail
   ARE exceptions to the no-fetch rule — they go through the parent CMS,
   not direct from the iframe.
✓  Each request* method requires its allow*: 'yes' toolParam. The admin
   configures this. Without it, the callback fires with an error —
   handle this gracefully in your code.
✓  All request* methods are asynchronous. Use callbacks, not return values.
✓  tool.requestAI: max 3 concurrent requests, 60s timeout, 100K char prompt
   limit. Prompts exceeding the limit are truncated; check the error param
   for a truncation notice.
✓  tool.requestObjects: max 500 operations per batch. Avoid batching
   operations that would exceed 10 MB total (Firestore writeBatch limit).
✓  Batch operations are ATOMIC — if one fails, the entire batch is rolled
   back. Design your batch workflows with this guarantee in mind.
✓  Unregistered mainObjectTypes auto-create a default folder on first use.
   You do NOT need the CMS admin to pre-register types for your tool to work.
✓  Emails sent via tool.requestSendEmail are automatically wrapped in the
   CMS-branded email template with header/footer/borders — you only provide
   the body content. The title is displayed prominently in the email header.
✓  tool.requestExportPdf exports as an HTML document that can be printed to
   PDF via browser Print → Save as PDF. Use attachments: [] in
   requestSendEmail to attach exported files to emails.
✓  Always produce complete, functional code — no placeholders, no TODO comments.
